# OIDC Integration Reference

Generic reference for integrating an app with authentik via OpenID Connect (Pattern 1 in
[`docs/app-integration-patterns.md`](app-integration-patterns.md)).

**Live worked example:** `../wordpress-ai-publisher` (Phase 4, ADR-011) uses this pattern against
this gateway ÔÇö issuer `https://<your-sso-domain>/application/o/<app-slug>/`, discovery document
confirmed reachable at deploy time (see `TODO.md` Phase 4 for the operator's own real values). Its
own OIDC client config/behavior is documented in
`../wordpress-ai-publisher/docs/AUTHENTICATION.md`, following exactly the checklist below.

## Generic OIDC Terms

| Term | Meaning |
|---|---|
| **Issuer URL** | The base URL authentik publishes as the OIDC issuer for a given provider, e.g. `https://auth.example.com/application/o/<app-slug>/`. Used for discovery (`/.well-known/openid-configuration`). |
| **Client ID** | Public identifier for the app's OIDC client, generated by authentik when the provider is created. |
| **Client Secret** | Confidential secret paired with the Client ID, used by confidential (server-side) clients to authenticate token requests. Never expose in frontend/browser code. |
| **Redirect URI** | The exact callback URL authentik redirects back to after login. Must match what's registered on the provider, character for character. |
| **Scopes** | Requested claim groups, e.g. `openid profile email groups`. |
| **Claims** | The actual identity data returned in the ID token / userinfo endpoint (e.g. `sub`, `email`, `groups`). |
| **Logout URL** | Where authentik redirects after an OIDC end-session request, if configured. |

## Recommended Scopes

- `openid` ÔÇö required, enables OIDC (vs. bare OAuth2).
- `profile` ÔÇö name and basic profile fields.
- `email` ÔÇö verified email address.
- `groups` ÔÇö group membership, if the app needs it for authorization (authentik must be
  configured to include a `groups` scope/claim).

## Example Redirect URI Pattern

```
https://app.example.com/oauth/callback
```

Replace with the app's actual callback path (varies per framework/library ÔÇö e.g.
`/auth/callback`, `/api/auth/callback/authentik`, `/oauth2/callback`).

## App Onboarding Checklist

1. In authentik: **Applications Ôćĺ Providers Ôćĺ Create Ôćĺ OAuth2/OpenID Provider.**
2. Set the exact redirect URI(s) the app will use.
3. Set scopes (`openid profile email`, plus `groups` if needed).
4. Create the matching **Application**, and grant access via group membership.
5. Copy the issuer URL, Client ID, and Client Secret into the app's own configuration (its own
   `.env`, not this repo's).
6. Confirm discovery works: `curl https://auth.example.com/application/o/<app-slug>/.well-known/openid-configuration`.
7. Test a full login Ôćĺ redirect Ôćĺ callback Ôćĺ token exchange cycle before relying on it.
8. Confirm logout behavior matches expectations (see per-app logout notes, Phase 5).

## Security Notes

- Client secrets are credentials ÔÇö store them the same way you'd store a database password
  (never in Git, never in client-side/browser code).
- Always use `https://` redirect URIs once anything leaves localhost.
- Prefer the `groups` claim for authorization decisions inside the app, rather than re-implementing
  role logic separately from authentik's group model.
- Validate the ID token's signature and `aud`/`iss` claims ÔÇö don't trust an unsigned/unverified
  token from any source, including reverse-proxy-injected headers (that's Pattern 3, a different
  trust model ÔÇö see [`docs/security.md`](security.md)).
